# 死锁

- 408 常考题
  是操作系统的核心概念，并发和多线程场景下的特有现象

就像两个人过独木桥，谁也不肯退让，结果谁也过不去
就是多个线程互相等着对方手里的资源

多线程 就像给程序开了多个分身，让它们处理不同类型的任务
比如在下载文件时。程序还可以流畅的响应你的点击操作，互不耽误

本质是 资源竞争 + 错误的资源获取顺序

- 线程A持有资源X，等待资源Y
- 线程B持有资源Y，等待资源X

形成条件

- 互斥条件，资源(进程分配) 一次只能被一个线程占用
- 请求与保持，已经拿到资源的线程 还在请求新资源
- 不可剥夺，资源不能被强制夺走
- 循环等待 线程之间形成了资源等待环

## 怎么解决

- 大家按顺序排队拿，或者时间过期就放松，别互相干耗着
- A的习惯：先拿1，再拿2， B的习惯：先拿2，再拿1
  A想干活，先拿1再拿2，B想干活，虽然想先拿2，但不允许,它也先拿1

如果是A抢到1，B就等着。A拿走2，干完活，1和2都释放了，B才能拿1，
如果是B抢到了1，A就等着，反正要有一方干完活，释放资源。

# stack_heap

## 首先，我想结合JS内存模型来回答

- 栈主要用于存储执行上下文，基本类型和函数调用，由系统自动管理。
- 堆主要用于存储对象，数组、函数等引用的复杂数据，由开发者(new 申请内存 delete释放内存)或GC(Garbage Collection 垃圾回收)管理
- 栈里存的是值或者复杂数据类型的引用地址 快
- 堆里存的是对象的真实数据 不连续，还有碎片

- 垃圾回收：标记清除、引用计数
  标记清除：打扫卫生，断舍离 贴个标签，没贴的就扔掉回收
  引用计数：有人使用就+1

- 性能差异 栈快(只需要挪动指针) 堆慢

- 内存
  栈是连续的内存(LIFO)
  堆不是连续的，随机分配，需要GC

- 闭包
  本该在栈释放的数据被引用到了堆中

## 数据结构的常客

- 栈是线性数据结构，LIFO
- 堆 有大顶堆，小顶堆 树状结构，非线性数据结构 排序

## 总结

- 栈和堆的核心区别在于存储内容和管理方式。
  栈主要用于存储执行上下文和基本数据类型，由系统自动分配和释放，效率很高。
  堆主要用于存储对象等引用数据类型，由垃圾回收机制进行管理
  在JS中，变量本身在栈中，而对象实际存在堆中，栈里存放的是指向堆的引用地址

- 栈的特点是内存连续，后进先出，所以访问速度快，而堆是动态分配的，可能产生碎片，访问相对慢
  从生命周期来看，栈中的数据随着函数执行结束自动释放，而堆中的数据只要引用就不会被垃圾回收
  会影响我们的浅拷贝、闭包以及内存泄漏等问题

## 怎么判断对象是否可回收

对象是否可回收取决于：是否还能被访问(是否可达)

```
function fn(){
let obj={a:1} // 会被回收
}
fn()
```

```
let globalObj;
function fn(){
    let obj={a:1} // 不会释放
    globalObj=obj
}
fn()
```

```
let a={}
let b={}
a.x=b
b.y=a
循环引用
a=null
b=null
```

```
let obj={a:1}
return function inner(){
  console.log(obj) // 不会被回收 闭包
}
```

## 什么是内存泄漏

本该被回收的内容，因被意外引用而无法释放，导致暂用持续增长，最终引发页面卡顿甚至崩溃。

- 定时器未清理
  react的组件卸载了，但是定时器没有取消
  定时器内部函数引用了外部变量 变量被长期引用
- 事件监听未被移除
- DOM的引用未释放
  let el=document.getElementById('app')
  document.body.removeChild(el)
  el=null // 要手动置为空
- 全局变量/意外挂载到全局
  var -> window,永远是root 可到达
- 闭包导致的内存泄漏
- Map/Set 使用不当
  Map和json的区别，obj可以作为key
  obj=null 由于Map对于他强引用，obj不会释放
  weakMap,WeakSet key obj 被回收 启动回收 弱引用关系
- 订阅发布者模式
  不取消订阅，一直引用
- Promise 一直不结束
- Ajax api请求发起了，但结果还未到达，组件卸载了
abort
