# 浏览器底层

- 当打开 Chrome 的时候，意味着什么？
  启动了一个主进程 process 唯一的 PID
  上网的代理程序
  分配资源的最小单位
  线程 thread CPU 调度和执行的基本单位

- 为什么是 chrome？
  chrome 优秀 市场占有率最高的浏览器
- 主要功能？
  子进程

  - html/css/js 渲染页面 render
  - v8 引擎 js 执行
    单线程 简单
  - 网络下载 network
    通信

- 多进程多线程架构

  - 主进程 browser
    负责管理和调度的

    - GPU 进程 显卡 渲染 3D
      animation transition tanslate3D
    - netework 网络进程
    - Storage 进程 本地存储 缓存 Localstorage Cookie 前端数据库

    - 标签页是独立的子进程
      - 渲染进程 render process 渲染引擎 Blink
      - JS 执行进程 v8 引擎
        JS 单线程（主） 运行起来了

- 进程
  分配资源的最小单位
  每个 tab 页是个独立的进程 内存开销大
  每个页面显示更快
  如果进程/线程 如果崩溃的话 所有页面都崩溃了
  IE 浏览器动不动就崩溃 -> 用多 tab 多进程来做隔离

- 单线程
  同步任务(sync)，异步任务(async)放到 event loop 中
- 多线程 并行处理
  线程共享进程的资源

- 当一个进程关闭之后，操作系统会回收进程所占用的内存
- 每一个进程只能访问自己占有的数据，不能访问其他进程的数据。如果崩溃了，也不会影响其他进程的运行。如果进程之间需要进行数据通信，需要通过进程间通信（IPC）机制来实现

## chrome 多进程架构

- 浏览器进程
  主要负责界面显示 （浏览器软件界面）、用户交互、子进程管理、同时提供存储功能
- 渲染进程
  核心任务是将 html/css/javascript 转换为用户可以与之交互的网页
  排版引擎 Blink 和 JS 引擎 v8 都运行在这里
  默认情况，chrome 会为每个 Tab 标签创建一个渲染进程
- GPU 进程
  早期其实是没有的，GPU 使用的初衷是为了实现 3D CSS 的的效果，
  网页、UI 都采用 GPU 来绘制，显卡的加速
- 网络进程
  负责页面的网络资源加载，独立的进程
- 插件进程
  flash、chrome extension
