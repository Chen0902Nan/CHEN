# Node.js 事件循环机制

1. 核心定义 (What)
   Node.js 的事件循环（Event Loop）是其处理非阻塞 I/O 的核心机制。因为 JavaScript 是单线程的，为了避免 I/O 操作（如文件读写、网络请求）阻塞主线程，Node.js 借助底层 Libuv 库，将耗时的 I/O 操作交给操作系统或线程池处理，待任务完成后，通过回调函数告知主线程。事件循环就是负责不断轮询这些回调并将其加入执行队列的机制。
2. 六个阶段的循环流转 (How)
   事件循环并非一次性执行完，而是通过一个 Tick（滴答）不断旋转，主要经历以下阶段：
   - Timers (定时器阶段)：执行 setTimeout 和 setInterval 的到期回调。
   - Pending Callbacks：处理系统级别错误（如 TCP 连接错误）。
   - Idle, Prepare：系统内部使用。
   - Poll (轮询阶段)：核心阶段。获取并执行新的 I/O 回调。如果没有其他任务，它会在这里阻塞等待 I/O 结果。
   - Check (检查阶段)：专门执行 setImmediate 的回调。
   - Close Callbacks：执行关闭操作的回调（如 socket.on('close')）
3. 关键点：微任务的“插队”规则 (The Secret Sauce)
   这是区分初级和高级工程师的关键。除了阶段性执行的“宏任务”，还有一个 微任务队列：
   微任务优先级：process.nextTick > Promise.then。
   执行时机：微任务不是等到事件循环全部跑完才执行，而是在每一个宏任务（阶段性回调）执行完毕后，立即清空微任务队列，然后再进入下一个阶段。
4. 高级进阶：必须掌握的两个避坑场景 (Pro Insights)
   如果面试官问“为什么要了解这个”，你可以补充这两个实战痛点：
   - 场景一：Poll 阶段的跳转
     在 I/O 回调内部，setImmediate 永远比 setTimeout 先执行。因为在 Poll 阶段之后，事件循环会紧接着进入 Check 阶段，而 setTimeout 必须等到下一轮循环的 Timer 阶段。
   - 场景二：微任务阻塞 (Microtask Starvation)
     process.nextTick 优先级极高。如果递归调用 nextTick，会产生“无限循环”，导致事件循环始终无法进入下一个阶段，造成 I/O 饥饿，最终阻塞整个 Node.js 服务。这也是为什么高性能代码应谨慎使用 nextTick，转而使用 setImmediate 的原因。

   - 面试官视角：你的回答亮点
     区分度：你把“宏任务”和“微任务”的执行时机解释清楚了（即宏任务间插队）。
     架构思维：你提到了 Libuv 和操作系统底层交互，显示了你对运行时环境的深刻认知。
     工程实践：你提到了“I/O 饥饿”和“递归阻塞”的问题，说明你写过高并发、健壮的代码，不仅仅是理论派。
     给你的“杀手锏”话术（结尾总结）：
     “总结来说，Node.js 的事件循环是一个动态调整的任务调度系统。它通过‘阶段化’保证了调度的有序性，通过‘微任务队列’保证了响应的及时性。但在开发中，我们必须通过合理控制异步任务的粒度，避免主线程在微任务或 CPU 密集计算中‘卡死’，这才是构建高可用 Node.js 应用的关键。”
