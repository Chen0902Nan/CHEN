# 从url输入到浏览器渲染完成

## 浏览器多进程架构

- 现代浏览器都是多进程架构
  - 主进程
  - 渲染进程
  - 网络进程
  - 插件进程
  - GPU进程
    多进程之间相互配合

## URL解析与DNS查询(网络层)

当我们输入url，按下回车后，浏览器首先做的就是解析URL

1. URL解析：浏览器会检查协议、域名、端口、路径。如果包含非法字符，会进行URL编码
2. DNS查询

- 浏览器缓存：首先检查浏览器的DNS缓存
- 系统缓存：查找操作系统的hosts文件
- 路由器缓存：向网关发起查询
- 递归查询：如过以上都未查询成功，就向ISP运营商DNS服务器请求。ISP从根域名服务器->顶级域名服务器(.com)->权威域名服务器，最终拿到IP地址

- DNS负载均衡
  我们每次访问同一个域名，但是每次响应的并非是同一个服务器，一般大公司会有成百上千台服务器来支撑访问，避免同时间的高并发问题。DNS可以返回一个适合的IP给用户，比如根据每台机器的负载量，该机器离用户地理位置的距离等等。

## 建立连接(TCP/TLS)

拿到IP后，浏览器需要与服务器建立通信

1. TCP三次握手
   SYN -> SYN-ACK -> ACK

- 第一次：客户端发送SYN(Seq=x)包给服务器，并进入SYN_SEND状态，等待服务器确认
- 第二次：服务器收到SYN包，必须确认客服的SYN，发送一个ACK包(ack=x+1)，同时自己也发送一个SYN包(Seq=y)，即SYN+ACK包，此时服务器进入SYN_RECV状态
- 第三次：客户端收到服务器的SYN包+ACK包，向服务器发送确认包ACK(ack=y+1)，此包发送完毕，客户端和服务器进入ESTABLISHED状态，完成三次握手

三次握手的主要目的是确认服务器和客户端之间都有发送和就收消息的能力。握手过程传送的包不包含数据，三次握手完毕后，客户端与服务器才正式开始传送数据。理想状态下，TCP连接一但建立，在通信双方中的任何一方主动关闭连接之前，TCP连接都将被一直保持下去
HTTP1.0 短连接 HTTP1.1 Keep-alive长连接

2. TLS握手(HTTPS)
   如果使用的是HTTPS，还需要经过TLS握手。客服端与服务端先进行非对称加密，后再使用对称加密进行数据传输

3. 四次挥手

- 第一次：客户端向服务器发送一个FIN包，告诉服务器我的数据已经发送完毕了，此时客户端还可以接受数据
- 第二次：服务器收到FIN包后，发送一个ACK包给客户端并且带上自己的序列号seq，确认序列号为收到的序列号+1。服务端进入关闭等待状态。TCP服务器通知高层的应用进程，客户端向服务器的方向就释放了,这时候处于半关闭状态，即客户端以及没有数据要发送了。服务端确认了客户端数据发送完毕的消息。
- 第三次：服务器发送一个FIN，用来关闭服务器到客户端的数据传输，告诉客户端我的数据也发送完毕，不会再给你发送数据了。
- 第四次：主动关闭方收到FIN，发送一个ACK给被动关闭方，确认序号为收到序号+1，此时，客户端就进入TIME-WAIT状态。当经过2\*MSL(最长报文段寿命)的时间后，客户端撤销相应的TCB后，进入CLOSED状态

四次挥手的过程的目的就是确认客户端和服务器都发送完了消息，双方发送的消息都接收完毕了。

## HTTP请求与响应(应用层)

连接建立后，浏览器发送HTTP请求

1. 发送请求：带上Cookie、User-Agent等Header信息
2. 服务器处理：服务器(Ngnix/后端)收到请求，解析路由，查询数据库，渲染摸版或返回JSON
3. 响应数据：浏览器收到响应体，检查状态码(200,301/302等),如果响应头有Content-Encoding:gzip 浏览器会解压数据

## 浏览器渲染引擎

这是最关键的环节，浏览器接收到 HTML 代码后，渲染进程（Renderer Process）开始工作：

1. 构建 DOM 树：HTML 解析器将 HTML 字符串解析成 DOM 树。遇到 <script/> 时会阻塞解析，除非加了 async 或 defer。
2. 构建 CSSOM 树：解析 CSS 文件，生成层叠样式表模型。
3. 合成 Render 树：将 DOM 树和 CSSOM 树结合，剔除不可见元素（如 display: none）。
4. Layout（布局/重排）：浏览器计算每个节点在页面中的精确位置和大小。
5. Paint（绘制）：将计算好的几何信息转换为屏幕上的像素点（填充颜色、阴影等）。
6. 合成（Compositing）：这是现代浏览器的高性能部分，将图层（Layer）交由 GPU 处理，减少 CPU 的重绘压力（will-change 属性就是为此而生）。

## 页面交互与脚本执行

1. js执行：在DOM树构建过程中，js会解析并执行脚本，可能会修改DOM或者CSSOM
2. 页面完成(Load)：当所有的资源(图片、样式、脚本)都加载完毕，会触发window.onload事件。
3. 重排与重绘：当你在JS中修改DOM或样式时，浏览器会触发重排或重绘
   对于DOM，避免频繁操作DOM,可先缓存节点或用文档碎片批量更新
   document.createDocumentFragment()

# OSI七层协议

OSI七层模型是计算机网络通讯的标准化框架。我们可以分为应用处理层、传输控制层、物理传输层三个维度来理解

1. 应用层

- 作用：直接为应用程序提供网络服务，是用户和网络的接口
- 例子：HTTP/HTTPS(网页传输)、FTP文件传输、SMTP邮件
- 比喻：就像外交部，负责向对方传达具体的业务意图

2. 表示层

- 作用：负责数据的表示，包括数据格式化、加密、解密、压缩与解压。确保发送端和接收端能看懂对方的数据
- 例子：SSL/TLS加密、JPEG/GIF图片编码、JSON/XML数据格式
- 比喻：翻译官、加密室，负责吧数据翻译成对方能懂的语言，或者进行加密以防止偷窥

3. 会话层

- 作用：建立、管理和终止应用程序之间的会话。它负责同步不同机器上的应用程序进程
- 例子：断点续传、Session ID维护、RPC(远程过程调用)
- 比喻：会议协调员，负责规定什么时候开始通话，通话过程中断了如何自动恢复

4. 传输层

- 作用：负责到进程(端到端)的通信。它是网络中实现可靠传输或高效传输的关键
- 协议：
  TCP：面向连接，保证可靠性(有重传机制、流量控制、拥塞控制)
  UDP：无连接，追求速度，允许丢包
- 比喻：快递公司，TCP是挂号信，必须确保送达，UDP是普通信件，快但丢了也没办法

5. 网络层

- 作用：负责主机到主机的通信，核心是寻址(IP地址)和路由选择(确定最佳路径)
- 例子：IP协议、ICMP(ping命令)
- 比喻：邮政分拣中心，通过IP规划路线，确定包裹是从哪个国家到哪个国家

6. 数据链路层

- 作用：负责相邻节点之间(比如你的电脑到路由器)的数据帧传输。核心是MAC地址寻址
- 例子：以太网协议、APP协议
- 比喻：交通警察，在同一个物理网络(局域网)内，确保数据包准确无误地从一个网卡跳到另一个网卡

7. 物理层

- 作用：将数据转化为比特流(0和1)，通过物理媒介(网线、光纤、电磁波)传输
- 例子：网线规格、光纤、集线器、电信号
- 比喻：管道工，负责铺设管道，保证电流或光信号能通道对方那里

虽然 OSI 有 7 层，但在实际的工业界中，我们更多讨论的是** TCP/IP 四层模型**（即：应用层、传输层、网络层、网络接口层）。因为 OSI 模型在理论上非常完整，但由于其实现过于复杂，导致在现代互联网中，应用层、表示层和会话层往往在同一个软件层面被合并处理了。这种简化也是互联网能够快速普及的核心原因之一。”
TCP/IP 是事实标准，OSI 是理论标准

TCP/IP 模型是互联网工业界实际通用的协议标准。为了方便描述，它将复杂的 OSI 七层模型简化为四层。这四层从上到下分别是：

1. 应用层 (Application Layer)
   职责：直接面向用户应用，定义数据格式和交互规则。
   协议：HTTP, HTTPS, DNS, FTP, SMTP 等。
   一句话总结：它是用户看到的页面或接口，负责把业务数据交给底层的传输层。
2. 传输层 (Transport Layer)
   职责：负责端到端的进程通信，提供可靠或快速的连接控制。
   协议：TCP（保证可靠性）、UDP（追求高效率）。
   一句话总结：它是数据的‘运输车队’，通过端口号（Port）精准找到目的地的应用程序。
3. 网络层 (Internet Layer)
   职责：负责主机到主机的寻址与路由选择，决定数据如何跨网络到达目的地。
   协议：IP（IPv4/IPv6）, ICMP 等。
   一句话总结：它是数据的‘导航系统’，通过 IP 地址规划路径，解决‘从哪儿来到哪儿去’的问题。
4. 网络接口层 (Network Interface Layer)
   职责：负责物理媒介的访问与数据链路的传输，将数据转化为比特流发向物理线路。
   协议：以太网协议、ARP 协议。
   一句话总结：它是数据的‘搬运工’，通过 MAC 地址实现局域网内的精准交付，并将数据最终转化为物理信号。”
