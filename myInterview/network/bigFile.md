1. 开场定性：问题的本质
   大文件上传不是一个简单的文件传输问题，而是一个涉及前后端协作的系统工程。”

痛点：首先要明确，传统上传在面对 GB 级文件时，会面临 HTTP 连接超时、服务器内存溢出、网络波动导致前功尽弃 等核心痛点，长时间的上传等待之后，如果失败，用户体验感会很差。

核心方案：我们的标准解法是 “分片上传 + Hash 校验”，辅以 “断点续传” 和 “秒传” 策略。

2. 前端架构：性能与流控
   “在前端，我关注的是计算效率和用户交互的流畅度。”

物理切片：利用 File.slice 对文件进行逻辑切割，这本身不占用额外内存。
指定每个切片的chunkSize:5*1024*1024，

指纹计算（关键点）：我会引入 Web Worker 在后台线程计算文件的 MD5。为了防止大文件读取导致 UI 卡顿，我会采用 分块读取、增量更新 的方式喂给 SparkMD5 库。
主要用的是 sparkMD5中的 spark.append(event.target.value)进行增量更新，利用FileReader实例化reader对象，通过reader.readAsArrayBuffer将每个chunk变成二进制数据，通过spark,append()对MD5进行增量更新

并发调度：为了避免一次性发出几百个请求导致浏览器挂起，我会手动封装一个 异步并发池。通过 Promise.race 维持一个固定数量（如 3-6 个）的上传窗口，实现流量削峰。

3. 断点续传与秒传逻辑
   “这是提升用户体验的杀手锏，通过预检接口实现。”

秒传：上传前发一个 check 请求。后端根据fs.existSync()校验MD5发现文件已存在，直接返回成功，用户体验极其惊艳。

断点续传：如果上传过部分，后端会返回已收到的 切片索引列表。前端通过 filter 过滤掉已存在的切片，只传剩下的部分，真正做到“从哪里断，从哪里续”，解决用户的一个使用痛点。

4. 后端处理：稳定性与 I/O 效率
   “后端的核心挑战是高并发下的文件一致性与内存管理。”

暂存机制：使用以文件 Hash 命名的目录暂存切片，文件名即索引。我推荐使用 Multer 中间件将数据流先行落地磁盘，确保内存安全。

排序：合并之前，为了防止切片之间的顺序混乱，我们一般会先用sort()对文件进行一个排序，防止合并的时候出现文件错乱导致数据丢失等问题

流式合并：合并阶段，我绝不会一次性读入所有切片。我会利用 Node.js Stream，通过 fs.createReadStream 和 fs.createWriteStream 进行管道流式写入。

安全性：合并后，后端会进行二次 MD5 校验，确保磁盘写入过程没有发生位翻转或数据丢失。

5. 总结与工程思考
   “除了功能实现，我还会考虑生产环境的健壮性。”

异常处理：我会为每个切片请求增加 自动重试机制（如重试 3 次）。

清理机制：针对用户放弃上传导致的“碎片垃圾”，我会配置 Cron Job 定时任务 定期清理过期的临时文件夹。

云端演进：如果是超大规模业务，我会考虑将存储层迁移到 OSS（对象存储），利用其分片上传 API 减轻应用服务器的 CPU 和带宽负载。
