## 浏览器缓存机制

加快页面打开速度，降低服务器压力

### 强缓存

浏览器不发请求给服务器，直接用本地缓存的加载资源，速度快
第一次打开慢，之后会很快

- 服务器除了返回资源，还有响应头信息,服务器会给前端一个过期时间，

  Expires:Wed,32 DEC 2026 23:59:00 GMT -> Http 1.0 依赖本地时钟，如果用户修改了电脑时间，缓存机制就会混乱
  Cache-Control: max-age=3600 相对时间 -> Http 1.1 以秒为单位，不受客户端时间漂移的影响

- 资源提前就更新了，怎么办?
  index.css?v122 index.css?v123
  直接改文件名

- 过期时间到了
  过期时间到了，请求一定要发送
  如果变了 拿到新内容
  如果服务器端资源没有更新，用原版本 304

### 协商缓存

当强缓存失效后,浏览器并不会立刻下载新资源，而是会向服务器发起一个询问请求
如果没变 304 Not Modified
如果变了，发送内容

- 修改时间
  服务器发送内容给浏览器，响应头 Last-Modified
  浏览器在协商请求时，带上 If-Modified-Sine
  If-Modified-Sine/Last-Modified Http 1.0
  Ngnix 负载均衡，但不同服务器的时间可能不同

- ETag/If-None-Match hash 计算有没有改变
  资源的唯一的指纹
  1. 首次请求，服务器响应头返回一个ETag,作为资源的唯一指纹
  2. 再次请求，当强缓存失败，浏览器会再次请求，并在请求头中带上 If-Node-Match，If-None-Match 其实就是上次的ETag
  3. 服务器判断If-None-Match和资源当前的ETag
     相同：304
     不同：200 + 新的资源
  - 优点：优先级高于Last-Modified，基于内容，更精准
  - 缺点：生成ETag会增加服务器的计算开销
