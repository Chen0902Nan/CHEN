## 常见的状态码及其含义

- 1xx 信息响应

1. 101 Switching Protocols：最典型的是在WebSocket握手时，客户端告知服务端切换协议，服务端响应 101，随后连接升级为全双工通信

- 2xx Success

1. 200：万能的成功
2. 201 Created：当你执行POST创建资源成功后，返回201能够明确告知客户端资源已经落地
3. 204 No Content：比如执行DELETE或某些更新操作，操作成功但不需要返回响应体，返回204可以节省带宽，也就是一种优雅地API设计
比如说CORS处理跨越时候，预检请求没有请求体

- 3xx 重定向 Redirection

1. 301：永久重定向。在SEO优化中很关键，搜索引擎会将旧链接的权重转移到新链接。
2. 302/307：临时重定向。常用于用户登录跳转。
3. 304 Not Modified：当客户端缓存了资源，服务端通过Header比较后，若资源没变，返回304，直接告诉浏览器“用你本地缓存的”，无需传输数据

- 4xx 客户端错误 Client Error

1. 400 Bad Request：语法错误或者参数校验失败，通常是客户端发来的请求格式不对
2. 401 Unauthorized：“未认证”，通常是没提供Token,或者是Token过期了
3. 403 Forbidden：这是拒绝访问，服务端知道你是谁，但客户端有登录信息却没有权限(比如普通用户去删除管理员账号)
4. 404 Not Found：资源不存在
5. 429 Too Many Requests：一般用于限流处理，告诉调用方：“你太快了，请求慢一点”

- 5xx 服务端错误(Server Error)

1. 500 Internal Server Error：程序崩了，代码抛出了未捕获的Exception,或者数据库挂了
2. 502 Bad Gateway：通常是Ngnix作为反向代理时，后端应用服务器过载或者正在停机维护
3. 503 Gateway Timeout：网关超时，比如Ngnix等了后端太久，后端还没给响应，通常是数据库查询慢或者发生了死锁

- 前端如何处理状态码问题
  让拦截器去处理“协议的共性401/500”，让 Service 层去解析“业务的逻辑400/422”，让 UI 组件只负责“渲染状态”。这才是工业级代码的样子。
